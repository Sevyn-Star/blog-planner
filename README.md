# Blog Planner

博客内容规划与思维导图可视化工具。你在 Markdown 里写主题，系统自动生成思维导图，帮你规划「写了什么、还要写什么、有没有重复」。

---

## 环境要求

- **Node.js** 18 或以上（推荐 20+）
- **npm**（随 Node.js 一起安装）

检查是否已安装：

```bash
node -v
npm -v
```

---

## 第一次使用

在项目根目录（也就是这个 `README.md` 所在的文件夹）打开终端，依次执行：

```bash
# 1. 安装依赖（只需第一次运行，或 package.json 变更后）
npm install

# 2. 启动项目
npm run dev
```

启动成功后，终端里会看到类似输出：

```
Blog Planner API running at http://localhost:3001
VITE ready
➜  Local:   http://localhost:5173/
```

**然后在浏览器打开：**

👉 **http://localhost:5173**

这就是 Blog Planner 的主界面。不要打开 3001，那是后端 API，给前端调用的。

## 多工作区（记录完全不同的内容）

Blog Planner 支持**多个独立工作区**，每个工作区有自己的思维导图、主题和层级，**互不影响、不用删除旧内容**。

### 切换 / 新建工作区

1. 看左侧边栏顶部的 **「工作区」** 下拉框
2. 选择已有工作区 → 界面切换到该工作区的导图和内容
3. 点 **`+`** 按钮 → 新建工作区（例如「技术博客」「读书笔记」「旅行计划」）
4. 创建后自动切换到新工作区，从空白开始规划

### 文件存在哪

```
workspaces/
├── default/              ← 默认工作区（旧版 topics/ 会自动迁移到这里）
│   ├── workspace.yaml    ← 工作区名称和描述
│   ├── taxonomy.yaml     ← 该工作区的层级树
│   └── topics/           ← 该工作区的所有主题
├── du-shu-bi-ji/         ← 另一个工作区
│   ├── workspace.yaml
│   ├── taxonomy.yaml
│   └── topics/
└── ...
```

每个工作区完全独立。在「技术博客」里写 Go 并发，在「读书笔记」里写完全不同的主题，切换下拉框即可，无需删任何东西。

> **说明：** 思维导图**只显示有主题的分类**。`taxonomy.yaml` 里预填的占位分类不会出现在导图上，只用于「覆盖度」页面的规划。新建工作区默认是空的 `{}`，不会再出现「默认 / 主题一 / 主题二」。

思维导图工具栏可切换布局方向：**↓ 自上而下**、**↑ 自下而上**、**→ 从左到右**、**← 从右到左**，每个工作区单独记住偏好。

## 怎么打开 / 关闭

| 操作 | 怎么做 |
|------|--------|
| **打开** | 终端里 `npm run dev`，浏览器访问 http://localhost:5173 |
| **关闭** | 在运行 `npm run dev` 的终端里按 `Ctrl + C` |
| **再次打开** | 进入项目目录，再跑一遍 `npm run dev` 即可，不用重复 `npm install` |

## 所有命令

在项目根目录的终端里运行：

| 命令 | 什么时候用 | 做什么 |
|------|-----------|--------|
| `npm install` | 第一次 clone / 下载项目后 | 安装依赖 |
| `npm run dev` | **日常开发、使用** | 同时启动后端 API（3001）和 Web 界面（5173） |
| `npm run sync` | 手动改了 `topics/` 里的文件，想立刻刷新图谱 | 重新扫描所有主题，生成 `graph.json` |
| `npm run check` | 写新主题前，想看看有没有重复 | 在终端输出相似/重复主题警告 |
| `npm run build` | 想打包前端静态文件 | 输出到 `dist/` 目录 |

> **注意：** 正常运行 `npm run dev` 时，改 `topics/` 里的文件会自动同步，一般不需要手动 `npm run sync`。

## Web 界面怎么用

启动后打开 http://localhost:5173，顶部有 5 个标签页：

### 1. 思维导图

- 展示你的博客层级结构和所有主题
- **紫色节点** = 分类（来自 `taxonomy.yaml` 或主题的 `paths`）
- **主题节点颜色：**
  - 灰色虚线 = 想法（还没开始写）
  - 橙色 = 有大纲
  - 蓝色 = 草稿
  - 绿色 = 已发布
- **橙色虚线箭头** = 主题之间的跨级链接
- 可以拖拽、缩放、用右下角小地图导航

### 2. 写作规划

- 列出所有**还没发布**的主题
- 按优先级和计划日期排序
- 顶部会显示**重复检测警告**（比如标题太像、同一路径下已有文章）
- 写新文章前先看这里，避免重复选题

### 3. 时间线

- **上半部分**：已发布文章，按发布日期从新到旧排列
- **下半部分**：进行中的主题（想法 / 大纲 / 草稿）

### 4. 覆盖度

- 表格展示每个层级路径下有多少主题
- 彩色条表示各状态占比
- 用来发现「哪个分类还空着、该补哪些题」

### 5. 新建主题

- 填表单即可创建新主题
- 创建后会自动生成 Markdown 文件到 `topics/` 目录
- 思维导图会**自动更新**，不用手动刷新页面

## 怎么添加 / 管理主题

有两种方式，选你习惯的：

### 方式 A：在 Web 界面新建（推荐入门）

1. 打开 http://localhost:5173
2. 点击顶部 **「新建主题」**
3. 填写：
   - **标题**（必填）
   - **层级路径**（从下拉选，或自己填，如 `技术/后端/Go/并发`）
   - **状态**：想法 / 大纲 / 草稿 / 已发布
   - **标签**、**优先级**、**计划日期**（可选）
4. 点「创建主题」
5. 切到「思维导图」tab，新主题已经出现

### 方式 B：直接写 Markdown 文件

1. 在 `topics/` 目录下新建 `.md` 文件，例如 `topics/my-new-post.md`
2. 复制 `templates/topic.md` 的内容作为模板
3. 修改 frontmatter（文件开头的 `---` 之间的 YAML）：

```yaml
---
id: my-new-post
title: 我的新主题
type: topic
status: idea          # idea | outline | draft | published
created: 2026-06-05
paths:
  - 技术/后端/Go/并发
tags:
  - go
priority: 2
planned_date: 2026-06-20
links:
  - id: go-channel-basics
    relation: related   # related | prerequisite | sequel | contrast
---
```

4. 下面写正文。正文里可以用 `[[go-channel-basics]]` 引用其他主题，会自动变成链接
5. 保存文件 → 如果 `npm run dev` 在跑，界面会自动更新

### 发布一篇文章

把对应主题文件的 `status` 改成 `published`，并加上 `published` 日期：

```yaml
status: published
published: 2026-06-05
```

保存后，时间线和思维导图里的节点颜色会自动变绿。

### 在 Web 界面编辑主题

- **写作规划 / 时间线**：点击主题卡片
- **思维导图**：**双击**节点
  - 紫色分类节点 → 修改层级路径
  - 主题节点（虚线/实线边框）→ 修改标题、状态、正文等

> 顶部黄色「重复检测」区域是系统自动生成的警告，**不可编辑**。要消除警告，需修改下方主题的标题或路径。

也可以继续直接编辑 `topics/` 下的 Markdown 文件，两种方式效果相同。

## 怎么定义博客层级

编辑项目根目录的 `taxonomy.yaml`：

```yaml
技术:
  后端:
    Go:
      - 并发
      - 性能
  前端:
    React:
      - 状态管理

生活:
  阅读:
```

- 嵌套 = 分类层级
- `- 并发` 这种数组项 = 最底层的分类
- 主题通过 `paths` 挂到任意路径，路径不必预先写在 taxonomy 里

改完保存，界面会自动刷新。

## 常见问题

**Q: 打开 5173 页面空白或报 API 错误？**  
A: 确认 `npm run dev` 两个服务都启动了。终端里应该同时有 API（3001）和 Vite（5173）的输出。如果只有 Vite，API 可能挂了，重启 `npm run dev`。

**Q: 改了 Markdown 但导图没更新？**  
A: 确保 `npm run dev` 在运行（它会监听文件变化）。或者手动跑 `npm run sync`。

**Q: 端口被占用？**  
A: 关掉占用 3001 或 5173 端口的其他程序，或重启终端再 `npm run dev`。

**Q: 详细的设计原理和 API 文档在哪？**  
A: 见 [docs/PROJECT.md](docs/PROJECT.md)。
